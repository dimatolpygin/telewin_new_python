# tgbot_py — Python-порт бота ИИ-поиска по прайсу «Домашний мастер»

Рабочий Python-порт бота (@NikaAi1Bot) — основной артефакт стенда `telewin_tests`.
Поиск = лексика (словарь+атрибуты+фаззи) ⊕ вектор (pgvector) через RRF.

**Отдельный изолированный git-репо** (ветка `dev`, БЕЗ remote — пуш некуда, только локальные
коммиты). Ведётся по okplan: `docs/STATUS.md` (снимок, читать первым), `docs/ROADMAP.md`
(критерии), теги `stage-N-done`.

- **Milestone 1 (этапы 0–10) закрыт** — гибридный поиск работает.
- **Milestone 2 «Апгрейд качества поиска» (этапы 11–16):** закрыты **11** (линейка+baseline),
  **12** (абстейн-гейт 34%→91%), **13** (словарь народных названий/кодов), **14** (атрибуты
  дюймы/цоколь/размеры), **15** (бренды — теги `stage-11..15-done`). Остался **16** (категорийные
  каналы для родовых + уточнение). Находки прочёса — `docs/NAHODKI_PRAJSA.md`, baseline — `docs/BASELINE_V2.md`.

## Неочевидное (без этого сломаешь или будешь искать час)

- **БД бота — отдельный pgvector-контейнер `telewin-test-pg` на :5434**, НЕ общий `postgres16`
  (:5432). `.env` → `PGPORT=5434`. Общий `postgres16` БЕЗ pgvector и делит хост с чужими
  проектами (`maria_doll`, `sbavito`, `landing_bot`, `hermes_agent`) — **его образ не менять**.
  Данные в volume `telewin_test_pgdata` (переживает пересоздание контейнера).
- Столбец эмбеддингов — **`embedding halfvec(3072)`**, не `vector`: HNSW по `vector` держит
  ≤2000 измерений, а gemini даёт 3072. Индекс `products_embedding_hnsw` (halfvec_cosine_ops).
- Эмбеддинги — OpenRouter `/embeddings` (`google/gemini-embedding-001`, 3072), нужен
  `OPENROUTER_API_KEY`. Пересчёт всех 11 864: `python -m bot.embed_index --all` — **~35 мин**
  (OpenRouter троттлит) и стоит денег. Текст = обогащённое имя (`bot/search/obogashenie.py`).
- **RRF-гаджет `bot/search/gibrid.py`: `W_VEC=0.3` — не поднимать.** При >0.3 вектор бьёт точный
  лексический матч и золотой набор регрессирует 38→37 («саморез черный 45»→ШУЦ). Порог 0.76
  держит абстейн на мусоре. Жёсткие каналы (штрихкод/артикул/производитель) короткозамыкают.
- Замер гибрида: `python -m bot.search.zamer_gibrid` (нужны посчитанные эмбеддинги).
  Результаты: слепой 85% top-1, золото 95% без регресса — `docs/ZAMER_GIBRID.md`.
- Слепой набор `data/test_nabor.json` — 302 записи; `ждём` = регэксп по имени товара,
  `ждём=null` = мусор (успех = абстейн). При росте набора валидировать регэксп против прайса.
- LLM-агент местами консервативен (уточняет до показа, «пассатижи»≠«плоскогубцы») — это промт
  этапов 3/6 (`bot/ai/agent.py`), НЕ поиск; тюнить отдельной задачей.
- **Слабое звено — агентский слой, НЕ поиск.** Баг «масло пихтовое: то нет, то есть» — LLM при
  `tool_choice=auto` относил реальный товар к «непрофильному» и отвечал «не найдено» БЕЗ вызова
  поиска. Фикс (коммит `af94e1e`): предохранитель в `run_agent` — на отказ без поиска форсирует
  повтор с `tool_choice=required` (+ `_zayavil_otkaz`, промт-запрет). **Не убирать** — иначе баг
  вернётся. Сам поиск детерминирован; «не найдено» без пустой выдачи инструмента = ошибка агента.
- **Абстейн-гейт (этап 12) — `_chuzhoy_domen` в `bot/search/gibrid.py` + `data/chuzhoy_domen.json`.**
  На запрос из ЧУЖОГО домена (косметика/еда/техника/одежда/зоо) гибрид отвечает `[], "чужой домен"`
  ДО всех каналов (утечки шли и через лексику `словарь+атрибуты`, не только вектор — порогом 0.76
  не закрывалось). Абстейн 34%→91%. **Инвариант при правке словаря: 0 ложных срабатываний на
  ЗОЛОТОМ+СЛЕПОМ товарных** (иначе режется recall) — прогонять `baseline_v2` после каждой правки.
  Приборы-омонимы («масляный обогреватель», «мясорубка электрич») — ФРАЗАМИ, не голым словом
  (иначе гейтит стоковый плёночный обогреватель / втулку мясорубки). Реальный ассортимент (мыло,
  белизна, масло пихтовое, репеллент) маркеров НЕ содержит и не гейтится — проверять при добавлении.
- **Поле `proizvoditel` загрязнено СТРАНАМИ**: «Россия» 4411, «Китай» 360, «Без бренда» 304,
  «ноль» 61 (+Германия/КНР) — не бренды. **Этап 15 (`data/brendy.json → не_бренд`)** исключил их
  из стем-индекса канала производителя (был баг «россия»→4411 товаров). Уточнение к прежней заметке:
  народные марки инструмента в новом полном прайсе ЕСТЬ, но лишь как аксессуары (нож к рубанку
  MAKITA, пилки BOSCH, буры METABO) — самих электроинструментов этих брендов нет. Поэтому «честный
  отказ» динамический: `Poisk.brend_iz_zaprosa` + пометка `примечание_бренд` в агенте срабатывают,
  когда бренд назван, но НИ ОДНА найденная позиция ему не соответствует («болгарка макита»→УШМ-аналоги
  + «Makita нет»). Буст бренда +3 внутри семейства («болгарка вихрь»→УШМ Вихрь) — с целевой дотяжкой
  брендовых семейств, т.к. УШМ дробится по модели. Мерочные наборы бренды не называют → регресса нет.
- **Буст токена `слово_в_имени` = 4.0** (этап 13, `search.py`, было 3.0). Это «пользователь назвал
  подтип» (филипс→PH, гипсокартон→ШСГД, масляная→ПФ-115) — должен бить общее слово. Не опускать:
  при 3.0 «масляная **краска**» уходит к семейству Краска (водоэмульсионки/Серебрянка) поверх Эмаль
  ПФ-115. **В `слово_в_имени` класть только слова-ПОДТИПЫ, не прилагательные-определения:** токен
  `светодиодный→светодиод` пришлось убрать — он бустил ЛЮБУЮ строку со словом «светодиод», и родовой
  «светильник светодиодный» уходил в Модуль вместо Светильника (регресс 89→87). «Светодиодная лампа»
  как следствие отдаёт лампу накаливания — контекстный выбор отложен на этап 16.
- **Дюймы `1/2` vs `1½` (15мм vs 40мм) — ЗАКРЫТО в этапе 14** (`attributes.py`): прайс пишет
  полтора как `1 1/2"` или `1" 1/2 (40)` (НЕ символом ½), целое ПЕРЕД дробью → атрибут «1 1/2»≠«1/2».
  Там же добавлены атрибуты: **цоколь** (E27/E14/E40 + GU10/GX53/G9/G4/G23/T8, нормировка Е→E),
  **цв_температура** (2700/4000/6500К + тёплый/нейтральный/холодный), **ток_а** (16/25/40А автоматов).
  **Цоколь из запроса матчим по `q.lower()`, НЕ по `norm`** — norm-гомоглиф латинскую `x→х` ломает «gx53».
  **Цв.температура (2700/4000/6500) при лампа-контексте НЕ идёт в артикул-канал** (4 цифры иначе
  короткозамыкали на товар с таким артикулом). Остаточно путаются точные размеры крепежа (М12→М10 в
  части кейсов, 4.8→4.2, уголок «40*40*80» где 3-е число = высота ребра) — размерная точность 91%.
- Инструменты UAT/аналитики: `python -m bot.search.probe_batch <файл_запросов>` (произвольные
  запросы через гибрид → JSON, для регрессии), `python -m bot.search.dump_group "<группа>"`
  (выгрузка среза прайса), **`python -m bot.search.baseline_v2 [--save]`** — baseline milestone 2
  по 5 сегментам (абстейн-точность / top-1 / hit@5 / размерная точн.) + контроль золота. ГЛАВНАЯ
  мерка апгрейда: прогонять после КАЖДОЙ правки словаря/поиска. `--save` пишет `docs/BASELINE_V2.md`
  (**это замороженный baseline этапа 11 — не перезаписывать бездумно**; результаты правок класть в
  отдельный `docs/ZAMER_*.md`, как `ZAMER_ABSTAIN.md` для этапа 12). Наборы: `data/abstain_nabor.json`
  (44 чужих, `ждём=null`), `data/razmer_nabor.json` (32 размерных с «ждём»+«анти»). Сегментация
  слепого — по полю `прием` (мусор/сленг/опечатка/родовое/размер). Текущий baseline: абстейн 91%,
  разговорный 83%, родовой 89%, размер 84%, размерный-точный 84%; золото 95%.
