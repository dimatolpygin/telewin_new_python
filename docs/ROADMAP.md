# Дорожная карта — Python-порт бота @NikaAi1Bot (ИИ-поиск по прайсу «Домашний мастер»)

11 этапов: 0–6 — от пустого каркаса до протестированного бота (закрыты), 7–10 — «v2 поиска»
(добивание качества по итогам этапа 6). Каждый этап заканчивается **демонстрируемым
результатом** (что можно потрогать руками).

Ключевая идея: **MVP-first**. Этап 1 = живой поиск из командной строки (уже ценность),
дальше наслаиваются БД, ИИ, Telegram, улучшения и тесты — без перепиливания ядра.

**Мотивация порта:** боевой TS-бот даёт 85% top-1; Python-эталон `poisk2.py` — 92%
(snowball-стеммер вместо `natural`). Плюс Python-стенд умеет два канала, которых нет в TS:
прямой поиск по **штрихкоду** (EAN 100% в новом прайсе) и recall-канал по **подгруппе Форы**
(категории 100%). Порт не переписывание — он структурно усиливает поиск.

**Хранилище:** Postgres (паритет с боевым ботом + задел под pgvector/семантику).
**Данные:** новый полный прайс `2026-07-18T11-29-30.xls` → `products.json` (11 864 позиции,
категории 100%, штрихкоды 100%).

# v2 поиска (этапы 7–10)

Добивание качества по итогам этапа 6. Принцип: перед поиском стоит LLM (агент уже
переформулирует запрос и отсекает не-ассортимент), поэтому движку нужна **точность по

держим ≥95% top-1) и слепой (`test_subagenty`, растим, без просадок по группам).

## Этап 7 — Материалы-в-имени + порог soft_has

**Цель**: запрос «по дереву/бетону/металлу» ведёт в верный материал (сейчас «сверло по
дереву» отдаёт бетонные свёрла — `дер.` (3 буквы) выпадает из мягкого совпадения).

**Что делаем**:
- расширить `data/materialy.json` (`слово_в_имени`): `по дереву/деревянный→по дер`,
  `по бетону→по бет`, `по металлу→по мет`, `по кафелю/плитке→по кафел` (+ по провалам);
- опустить порог `soft_has` для сокращений с точкой (`дер./бет./мет.`), не ломая длинные
  (`оцинк.`↔`оцинкованный`).

**Критерии приёмки** (закрытие → tag `stage-7-done`):
- [x] «сверло по дереву» → top-1 «Сверло по дер.» (не бетон); «диск по металлу», «коронка по бетону/кафелю» — верный материал
- [x] слепой набор: «Инструмент ручной» top-1 21→22/29 (72→76%), ни одна группа не просела; общий 110→111/142 (77→78%)
- [x] золотой набор держит ≥95% top-1 (38/40, без изменений — регресс-гард)

**Как сделано (отклонение от плана):** вместо глобального опускания порога `soft_has`
до 3 симв (риск ложных `бита↔битум`, `мет↔метр` по всему скорингу) реализован адресный
механизм **«материал-в-имени»** в `data/materialy.json` (секция `материал_в_имени`):
стем запроса (`дерев/бетон/металл/кафел/плитк/камен`) → регэксп материала над именем,
ловящий обе формы предлога прайса (`по дер.` и `п/дер.`). В скоринге строка с нужным
материалом бустится (+4), с конфликтующим — штрафуется (−2). Устойчив к падежам и
не трогает общий `soft_has` (оттого золотой без просадки). Побочно починился слепой
«набор свёрл по металлу» (был top-1 «Набор щеток» → стал «Набор сверл п/мет.»).

## Этап 8 — pg_trgm фаззи-канал (опечатки)

**Цель**: опечатки не роняют поиск («малоток», «выключетель», «сместитель»).

**Что делаем**:
- триграммный канал (`pg_trgm` в Postgres по имени товара, либо offline `rapidfuzz` для json-прогона);
- канал нечёткого совпадения слитый в скоринг (fusion), НЕ замена лексики; жёсткие каналы
  (штрихкод/артикул/размер) сохраняют приоритет.

**Критерии приёмки** (закрытие → tag `stage-8-done`):
- [x] «малоток»→Молоток, «выключетель»→Выключатель, «сместитель»→Смеситель (в top-3)
- [x] приём «опечатка» на слепом наборе растёт: top-1 4→5/6 (67→83%), top-3 5→6/6
- [x] золотой ≥95% top-1 (38/40, без изменений), без регрессии по группам слепого (Инструмент 76→79%)

**Как сделано:** офлайн-фаззи-канал `bot/search/fuzzy.py` (`word_sim` = max триграммного
pg_trgm-сходства и нормализованного Левенштейна — второе точнее на коротких словах:
«малоток»~«молоток»=0.86 против тригр. 0.46). В `search.py` метод `fuzzy_semeystva`:
слово запроса ВНЕ словаря стенда (OOV-гард по `self.vocab` — всё из имён товаров) сверяется
с ИМЕНЕМ/каноном семейств; совпадение по имени-ключу весит выше синонима («выключетель»→
Выключатель, не «Блок»). Инжект всех семейств ≥0.8 как fusion-кандидатов (жёсткие каналы
штрихкод/артикул — приоритет, отрабатывают раньше). OOV-слово-опечатка исключено из `utochn`
(иначе префикс «выкл» бустил бы соседей). Побочно: тай-брейк +0.4 за полностью названное имя
семейства («ножовка»→«Ножовка», не «Ножовка-ручка») — дал общий прирост слепого 78→80% top-1.
Боевой путь — тот же триграммный `pg_trgm` в Postgres; офлайн-метрика стенда с ним в паритете.

## Этап 9 — Головное слово + длины см/м

**Цель**: различать головное слово от зависимого и мерить длину в см/м.

**Что делаем**:
- разбор головного слова запроса: «муфта **для трубы** ПП» → head=муфта (сейчас иногда
  отдаёт *Труба ПП*); «рулетка 3 метра» → head=рулетка, длина 3м, а не ширина 16мм;
- добавить в `razobrat` длины см/м (с приоритетом головного размера над шириной).

**Критерии приёмки** (закрытие → tag `stage-9-done`):
- [x] «муфта для трубы ПП 20» → top-1 «Муфта 20*1/2 ПП» (не Труба); «уголок/кран для трубы» — верное семейство; фитинги (Сантехника) 14→15/24
- [x] «рулетка 3 метра»→Рулетка 3м (не 2м), «шланг для душа 175 см»→175 см — верный типоразмер
- [x] золотой ≥95% top-1 (38/40, без изменений); слепой 113→115/142 (80→81%), ни одна группа не просела

**Как сделано:** (1) `attributes.py` — длины м/см в `razobrat`, нормировка в мм («3м»→3000,
«175см»→1750), головной размер приоритетнее ширины (стоит ДО разбора «*»/мм). Не хватаем число
после «*/х» (ширина пары «45*160см» → в диаметр*длина) и берём первое число диапазона «50/60 м».
(2) `search.py` — `head_dep`: слово до предлога «для/под» = головное (что ищем), после = зависимое
(с чем совместимо). В скоринге семейство=head → +2.5, семейство=лишь зависимое («для трубы») → −2.5.
Лечит «муфта для трубы ПП»→Труба. Регресс-контроль: длины сперва задели штору «45*160см» и шпагат
«L 50/60 м» (−2 группы) — исправлено уточнением регэкспа, итог без просадок.

## Этап 10 — Гибрид pgvector + RRF

**Цель**: семантический канал ловит композицию и открытые синонимы, которых нет в словаре.

**Что делаем**:
- эмбеддинги имён товаров (обогащённый текст: «Сверло по дереву перовое» вместо «по дер.»),
  колонка `vector` + индекс HNSW в Postgres (pgvector);
- векторный канал (эмбеддинг запроса → kNN) слит с лексикой через **RRF** (Reciprocal Rank
  Fusion); жёсткие каналы (EAN/артикул/точный размер) — приоритет, вектор их не размывает;
- замер «лексика vs гибрид» на расширенном слепом наборе.

**Критерии приёмки** (закрытие → tag `stage-10-done`):
- [x] слепой набор дорощен до ≥300 запросов (сквозная задача v2) — как приёмка метрики → **302** (279 товарных + 23 мусорных)
- [x] «болгарка»→УШМ и др. открытые синонимы ловятся вектором; жёсткие каналы (штрихкод/артикул) не деградируют → пассатижи→плоскогубцы, болгарка→УШМ и др.; штрихкод/артикул короткозамыкают
- [x] гибрид даёт top-1/top-3 строго выше чистой лексики на расширенном наборе; таблица «лексика vs гибрид» в `docs/` → слепой 81→85% top-1 (+11), 90→92% top-3; [`ZAMER_GIBRID.md`](ZAMER_GIBRID.md)
- [x] золотой ≥95% top-1, без регрессии → 38/40 = 95%, идентично лексике (без регресса)

# Milestone 2 — Апгрейд качества поиска (этапы 11–16)

**Мотивация.** Дорожная карта 0–10 дала рабочий гибрид, но UAT живым покупательским
языком (4 сегмента × 20 запросов + баг «масло пихтовое») вскрыл потолок: товарный top-1
≈77% на живом языке (против 95% на золотом — «удобном» наборе), а **честный абстейн лишь
60%** — на 40% чужих запросов поиск выдумывает семантически близкий товар. Прочёс всего
прайса 5 субагентами Haiku по группам дал конкретные списки под починку (сводка находок —
источник наполнения словарей и тестов). Инвариант всех этапов: **золотой набор ≥95% top-1
без регресса**.

Цели по сегментам (baseline живого UAT → цель): абстейн 60%→≥90%, разговорный 70%→≥82%,
технический 75%→≥85%, родовой 65%→≥78%.

## Этап 11 — Мерная линейка: расширенный тест-набор + baseline

**Цель**: сделать апгрейд измеримым. Без честной линейки нельзя доказать прирост и поймать регресс.

**Что делаем**:
- дорастить наборы запросами из находок субагентов: народные названия/сленг, точные размеры
  (дюймы 1/2 vs 1½, М10/М12, саморез 4.8), бренды, и отдельный АБСТЕЙН-набор чужих запросов
  («шампунь для волос», «масло для волос», «зубная щётка», «телевизор»);
- каждый товарный кейс с «ждём»-регэкспом (валидировать против прайса), мусорный — `ждём=null`;
- зафиксировать baseline по сегментам через `probe_batch`/`zamer_gibrid`, таблица в `docs/`.

**Критерии приёмки** (закрытие → tag `stage-11-done`):
- [x] абстейн-набор ≥40 чужих запросов, размеченных `ждём=null` (`data/abstain_nabor.json`, **44**); размерный набор ≥30 кейсов с точным «ждём» (`data/razmer_nabor.json`, **32**, 20 с «анти»-размером, все валидированы против прайса)
- [x] прогон даёт зафиксированный baseline по 5 сегментам (top-1/hit@5/абстейн) в `docs/BASELINE_V2.md` (`python -m bot.search.baseline_v2 --save`)
- [x] золотой набор без изменений (контроль: 38/40 = 95% top-1, идентично Milestone 1 — линейка код поиска не трогала)

**Baseline (гибрид, порог 0.76):** абстейн **34%** (23/67, риск №1), разговорный 83% top-1, родовой 89%,
размер-слепой 84%, размерный-точный 84% top-1 / размерная точн. 84%. Ключевая находка: утечки
абстейна идут через каналы `словарь+атрибуты+вектор` (не только вектор) → этап 12 не решается
одним порогом. Детали и списки утечек/промахов — `docs/BASELINE_V2.md`.

## Этап 12 — Абстейн (риск №1)

**Цель**: перестать выдумывать товар на чужой запрос; честное молчание 60%→≥90% без потери recall.

**Что делаем**:
- гейт «доменного пересечения» перед чисто-векторной выдачей (сейчас порог 0.76 пускает мусор);
- различение ЧУЖОГО квалификатора: «для волос/тела/лица/кожи» → не наш, при этом реальный
  ассортимент («мыло», «белизна», «масло пихтовое», «репеллент») продолжает находиться;
- маркировка/понижение конкретных ловушек хвоста (Шампунь д/моек, Щётка электр. графитовая и др.).

**Критерии приёмки** (закрытие → tag `stage-12-done`):
- [x] абстейн-точность на мусор-наборе (этап 11) 34%→**91%** (61/67, ≥90); hit@5 товарного recall не ниже baseline (разговорный 85%, родовой 95%, размер 94% — без изменений)
- [x] «шампунь для волос», «масло для волос», «зубная щётка электрическая», «телевизор самсунг» → «не найдено» (канал «чужой домен», проверено probe_batch)
- [x] «мыло», «белизна», «масло пихтовое», «средство от комаров» → находятся (Мыло хозяйственное / Белизна / Масло пихтовое 50мл / Средство от комаров Цифокс)
- [x] золотой набор ≥95% top-1 без регресса (38/40, идентично этапу 11)

**Как сделано:** доменный пре-гейт `_chuzhoy_domen` в `bot/search/gibrid.py` + словарь чужого
домена `data/chuzhoy_domen.json` (квалификаторы «для волос/тела/лица», зубная, сливочное… +
чужие товары телевизор/холодильник/велосипед/корм…). Гейт ПЕРЕД всеми каналами (утечки шли и
через лексику — одним порогом вектора 0.76 не закрыть). Валидирован: 0 ложных на золоте+слепом
товарных. Замер before→after — `docs/ZAMER_ABSTAIN.md`. 4 из 6 остаточных «утечек» — реальный
ассортимент (мусор-мислейбл этапа 6), т.е. фактическая точность выше 91%.

## Этап 13 — Словарь народных названий, синонимов и кодов

**Цель**: разговорный слой и коды прайса. Разговорный top-1 70%→≥82%.

**Что делаем**:

  экономка→люминесц., американка→разъёмное, масляная→эмаль ПФ-115, водоэмульсионка→В/Д;
- маппинг кодов→человеческие слова: крепёж (ШСГД/ШУЖ/ШС КРОВ), ЛКМ (ПФ/В/Д/ГФ), кабель (ВВГп/ПВС/NUM);
- словарь надёжнее вектора там, где он сейчас плавает («пассатижи»→гвоздь нестабильно).

**Критерии приёмки** (закрытие → tag `stage-13-done`):
- [x] «пассатижи»→Плоскогубцы, «экономка»→Лампа СД ЭКОНОМКА, «масляная краска»→Эмаль ПФ-115 — top-1 (probe)
- [x] коды крепежа/ЛКМ/кабеля с человеческих слов: «саморез для гипсокартона»→ШСГД, «саморез кровельный»→ШС КРОВ, ВВГп/ПВС

**Как сделано:** народные названия → синонимы существующих семейств в `slovar_svodnyy.json`

## Этап 14 — Атрибутный парсер: дюймы, цоколь, точные размеры, материал

**Цель**: технический top-1 75%→≥85%; перестать путать размеры.

**Что делаем**:
- дюймы `1/2` vs `1½` (это 15мм vs 40мм — стеммер их слепляет), плюс 3/4, 1", 1¼;

- приоритет ПОЛНОГО числового совпадения (М12≠М10, 4.8≠4.2); уголок «40*40*80» — 3-е число = высота ребра;
- материал как жёсткий фильтр: ПП/ПЭ/м-п/сталь/латунь невзаимозаменяемы.

**Критерии приёмки** (закрытие → tag `stage-14-done`):
- [x] «полтора дюйма 40»→Кран шар 1" 1/2 (40), «кран водоразб. 1/2»→1/2 (не 1½), «болт м12»→М12; размерная точность **91%** (29/32 ≥90%)
- [x] «лампа E27»→E27, «тёплый 2700»→2700К, «холодный 6500»→6500К, «нейтральный 4000»→4000К (все 9 лампа-кейсов OK)

**Как сделано:** расширен парсер `attributes.py` (цоколь E27/E14/E40+GU10/GX53/G9/G4/G23/T8,

из запроса в `atributy_zaprosa` (цоколь по `q.lower()` мимо гомоглифа x→х, температура/цоколь
из `sleng`). Гард: цв.температура (4 цифры) не короткозамыкает артикул-канал при лампа-контексте.
Скоринг новые атрибуты подхватывает сам (+4/−1.5). Остаточные 3 промаха (болт мебельный, саморез
4.8, металл 4.2 со спорным ждём) — вне цели, см. замер. Детали — `docs/ZAMER_ATRIBUTY.md`.
